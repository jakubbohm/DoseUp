Overnight 2026-07-18 I built throwaway PoCs under `spikes/` (fully isolated: own Directory.Build/Packages.props, not in DoseUp.slnx, main build unaffected) answering the three M0 `spike` issues. Verdicts:

- **#50 (Wolverine on net11 preview + ASB Basic, per-module outbox in own schema): GO — now confirmed against a REAL ASB Basic namespace (2026-07-18), not just the emulator.** Proven end-to-end locally — 4/4 integration tests green against the ASB emulator — AND smoked against a real Basic namespace (`doseup-spike-sb-basic`, rg `doseup-spikes`, Jakub's personal sub): unchanged PoC (AutoProvision off, SystemQueuesAreEnabled(false), queue-only) did a clean send→receive round-trip in ~2s, zero topic/session/system-queue/MessagingEntityNotFound errors; Wolverine keeps its reply endpoint in-memory (`stub://replies/`) so it needs no ASB system queue (why Basic suffices). Secret-free proof at `spikes/messaging-poc/evidence/asb-basic-real-namespace.txt`. Each module keeps its **own outbox AND inbox in its own Postgres schema** (`MessageStoreRole.Ancillary` + `Enroll<TDbContext>()`), single Main store in a `wolverine` schema; envelope joins the module transaction; EF Core 11 preview.5 runtime compat confirmed.
- **#51 (Wolverine × ASB emulator in Aspire harness): GO.** `RunAsEmulator()` + AutoProvision-off round-trip works; stub-transport fallback not needed.
- **#62 (Entra app registration from pipeline): CONDITIONAL GO — spike answered; NOT finished live.** GA Microsoft Graph Bicep extension; `az bicep build` compiles the tenant-scope Graph template offline. Implementation moved to a new issue **#75** (Entra External ID identity infra) with bootstrap sub-issues **#76–#79**. Two former open risks since resolved in our favour (verified against MS Learn 2026-05, via a research workflow): **GitHub OIDC into a CIAM tenant IS documented-supported** ("same as workforce" in the External Tenant Features doc), and **customer-app admin consent IS IaC-able** via `oauth2PermissionGrants`/`preAuthorizedApplications` (only the deploy identity's own bootstrap consent stays manual). Graph-Bicep at tenant scope against a subscription-less CIAM tenant is an explicitly documented GA scenario.

**#93 (EFCore.NamingConventions 10.0.1 on our EF 11 previews): GO — 2026-07-19, `spikes/snake-case-naming`.** 11/11 TUnit tests green on the exact root pins (EF 11.0.0-preview.5 + Npgsql EF 11.0.0-preview.5); schema applied to a real `postgres:17` container with every identifier unquoted (tables, columns, owned-type columns, PK/AK/FK, indexes, acronyms like `IANATimeZoneId` → `iana_time_zone_id`), and LINQ translation agrees with the design-time model. Three authoring rules **#42 must follow**, all silent failures otherwise, because the package renames only names it *generates* — never an explicit name, never raw SQL: (1) author module schema names lowercase (`HasDefaultSchema("membership")`); (2) author check constraints snake_case in **both** name and expression — the idiomatic PascalCase spelling emits DDL Postgres **rejects** (`column "DailyDoseLimit" does not exist`), and [[doseup-project-state]]'s domain-rule backstop maps violations by constraint name; (3) rename EF's own history table via `.MigrationsHistoryTable("__ef_migrations_history", schema)` or it stays a quoted `"__EFMigrationsHistory"` with snake_case columns inside. **The GO has an expiry date:** the package's dependency is a bounded range `[10.0.1, 11.0.0)`, not a floor — today's previews only satisfy it because SemVer sorts prereleases below their release, so bumping EF to **11.0.0 stable** raises NU1608, which the root's `TreatWarningsAsErrors=true` turns into a hard restore failure in `src/`. Loud and on a bump we control; fallback is still the ~20-line hand-rolled `IModelFinalizingConvention`.

Two Wolverine wiring findings the real M0 change must bake in (**now recorded in ADR-0002** at the reserved async-seam design note): (1) 6.20 removed the runtime Roslyn compiler from core — need `WolverineFx.RuntimeCompilation` OR static codegen (ADR-0002 already wants static codegen); (2) `UseEntityFrameworkCoreTransactions()` is NOT enough — must also call `Policies.AutoApplyTransactions()` or handlers never save and events bypass the outbox.
