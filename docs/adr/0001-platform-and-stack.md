# ADR-0001: Platform and stack

**Status:** Accepted · **Date:** 2026-07-13 · **Decided by:** Jakub (interview 2026-07-12/13) · **Amended:** 2026-07-13 — database: Cosmos DB → Neon Postgres (PRE-2) · 2026-07-13 — data access, async messaging, HTTP confirmation (PRE-4) · 2026-07-13 — reminder triggers + transport consolidation (PRE-3)

## Context

Greenfield hobby project, solo developer working with Claude. Azure is both the deployment target and part of the learning/showcase goal ([vision](../product/vision.md) G3/G4). Scale is an invite-only circle (≤ 50 accounts), which frees the stack to prioritize learning value and showcase quality over conservatism.

## Consequences

- **Preview risk is structural:** tooling (IDE, analyzers, CSharpier, TUnit, FastEndpoints, EF Core/Npgsql previews, Wolverine) may lag or break on a given preview. Mitigations: upgrades are isolated PRs (trivially revertable); M0 includes FastEndpoints- and Wolverine-on-net11 spikes; fallback is pinning the last working preview (EF Core 10 GA is the data-access pin).
- **Union caveats (as of 2026-07):** values stored as `object?` (value types box — fine at our scale, watch hot paths); "member providers" not yet implemented; early previews required a `UnionAttribute`/`IUnion` polyfill — check whether the current preview still does.
- Postgres: real relational **migrations, forward-safe from day one** (expand/contract, ADR-0004) via EF Core migrations (PRE-4); the schema baseline lands in M1 design; local dev/CI runs a plain Postgres container via Aspire's Postgres integration (ADR-0003); the Neon branching strategy is explored in PRE-9.
- **Neon is a non-Azure dependency (facts as of 2026-07):** the Azure Native Neon integration is retired and Neon's Azure regions are deprecated — new projects are AWS-only (nearest: Frankfurt/London). DoseUp therefore runs cross-cloud: ACA (Azure) ↔ Neon (AWS EU). Accepted deliberately; the NFR-2 latency budget is validated in M0, and the connection string is a real secret with no managed-identity path — held in ACA secrets/Key Vault (NFR-4).
- Scale-to-zero vs NFR-3 (reminders must fire regardless): resolved by PRE-3 — **storage-queue visibility alarms** wake the app exactly when a dose is due, so nothing polls while idle. The queue is a dumb alarm clock; the schedule aggregate stays the single source of truth: every fire is validated against current state (stale alarms are discarded), edits eagerly re-arm the new next occurrence via the outbox, and revocation is an optimization rather than a correctness requirement. Delivery latency ≈ KEDA poll + cold start, under a minute past the scheduled instant. Fine-grained design lands in M2. *(This supersedes PRE-4's blanket "scheduled messages rejected" note, which was argued from the RabbitMQ plugin's constraints — the pattern is sound when fire-time validation guards it.)*
- Messaging topology (PRE-4/PRE-3): at replica 0 nothing consumes — KEDA scalers wake ACA (azure-servicebus scaler on integration-event queues; azure-queue `visibleonly` scaler on reminder alarms). Delivery is at-least-once — consumers must be idempotent (Wolverine inbox). The scale-to-zero chain holds end-to-end: ACA at 0 → no durability polling → Neon suspends. Cost outlook at circle scale ≈ **€0/month** (Neon + ACA free grants; ASB Basic + Storage Queue ops in cents) — OQ-1 validates at the first bill.
- Vendor lock-in to Azure is accepted deliberately for hosting/auth (it *is* the showcase); the database is the deliberate exception — standard Postgres on Neon stays portable (any Postgres host can take a dump/restore).
- Polyglot toolchain (dotnet + npm) in every pipeline.
