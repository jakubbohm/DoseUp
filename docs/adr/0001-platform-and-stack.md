# ADR-0001: Platform and stack

**Status:** Accepted · **Date:** 2026-07-13 · **Decided by:** Jakub (interview 2026-07-12/13) · **Amended:** 2026-07-13 — database: Cosmos DB → Neon Postgres · 2026-07-13 — data access, async messaging, HTTP confirmation · 2026-07-13 — reminder triggers + transport consolidation · 2026-07-13 — TS contract-client pipeline · 2026-07-14 — authorization model · 2026-07-14 — deployment model (hand-authored Bicep stacks), Neon branching role, destructive-migrations policy (see [ADR-0004-delivery-and-process](0004-delivery-and-process.md)) · 2026-07-14 — domain enumerations (SmartEnum); feature-handler refinement of the "endpoint = handler" shorthand; validation runs in handlers; domain-rule model (`RuleViolations → 409`) · 2026-07-15 — c001: union caveats resolved on .NET 11 preview 6 (real grammar, generated-member reality, no polyfill) · 2026-07-18 — frontend styling: Tailwind CSS v4 (design workflow — [conventions/design.md](../conventions/design.md)) · 2026-07-19 — UI behavior primitives: Radix Primitives behind an owned-component wall (Base UI the named fallback); platform floor re-set to latest iOS/Android (NFR-1 — drops ≥ 16.4)

## Context

Greenfield hobby project, solo developer working with Claude. Azure is both the deployment target and part of the learning/showcase goal ([vision](../product/vision.md) G3/G4). Scale is an invite-only circle (≤ 50 accounts), which frees the stack to prioritize learning value and showcase quality over conservatism.

## Consequences

- **Preview risk is structural:** tooling (IDE, analyzers, `dotnet format`, TUnit, FastEndpoints, EF Core/Npgsql previews, Wolverine) may lag or break on a given preview. Mitigations: upgrades are isolated PRs (trivially revertable); M0 includes FastEndpoints- and Wolverine-on-net11 spikes; fallback is pinning the last working preview (EF Core 10 GA is the data-access pin).
- **Union caveats (resolved at c001 on preview 6, 2026-07-15):** values stored as `object?` (value types box — fine at our scale, watch hot paths). The polyfill question is closed — `UnionAttribute`/`IUnion` ship in the framework (since preview 5, reconfirmed preview 6). Real grammar: a union declaration takes a **mandatory parenthesized case-type list**; case types may nest inside the union body but the case list must qualify them (`public union RuleCheck(RuleCheck.Pass, RuleCheck.Fail) { … }`). Exhaustiveness is compiler-tracked (a missing switch arm is CS8509 → a build error under `TreatWarningsAsErrors`), but `==`/`Equals`/`GetHashCode` are **not** generated — the SharedKernel unions hand-write them. Preview 6 additionally allows non-public single-parameter case constructors and applies `not` patterns to the union value itself. A static member cannot share a nested case type's name, so case values construct as `new Result.NotFound()` — doc sketches' value shorthand is directional.
- Postgres: real relational migrations via EF Core, applied by a CD-run **migration bundle** — never at app startup; migrations **may be destructive** — deploys carrying them take a maintenance-window recreate (the expand/contract mandate was dropped — [ADR-0004-delivery-and-process](0004-delivery-and-process.md)); the schema baseline lands in M1 design; local dev/CI runs a plain Postgres container via Aspire's Postgres integration (ADR-0003); Neon branches serve as instant pre-deploy restore points only — no dev/CI branching — plus a scheduled `pg_dump` to Azure Blob for vendor-independent DR ([ADR-0004-delivery-and-process](0004-delivery-and-process.md)).
- **Neon is a non-Azure dependency (facts as of 2026-07):** the Azure Native Neon integration is retired and Neon's Azure regions are deprecated — new projects are AWS-only (nearest: Frankfurt/London). DoseUp therefore runs cross-cloud: ACA (Azure) ↔ Neon (AWS EU). Accepted deliberately; the NFR-2 latency budget is validated in M0, and the connection string is a real secret with no managed-identity path — held in ACA secrets/Key Vault (NFR-4).
- Scale-to-zero vs NFR-3 (reminders must fire regardless): resolved 2026-07-13 — **storage-queue visibility alarms** wake the app exactly when a dose is due, so nothing polls while idle. The queue is a dumb alarm clock; the schedule aggregate stays the single source of truth: every fire is validated against current state (stale alarms are discarded), edits eagerly re-arm the new next occurrence via the outbox, and revocation is an optimization rather than a correctness requirement. Delivery latency ≈ KEDA poll + cold start, under a minute past the scheduled instant. Fine-grained design lands in M2. *(This supersedes the earlier blanket "scheduled messages rejected" note, which was argued from the RabbitMQ plugin's constraints — the pattern is sound when fire-time validation guards it.)*
- Messaging topology (decided 2026-07-13): at replica 0 nothing consumes — KEDA scalers wake ACA (azure-servicebus scaler on integration-event queues; azure-queue `visibleonly` scaler on reminder alarms). Delivery is at-least-once — consumers must be idempotent (Wolverine inbox). The scale-to-zero chain holds end-to-end: ACA at 0 → no durability polling → Neon suspends. Cost outlook at circle scale ≈ **€0/month** (Neon + ACA free grants; ASB Basic + Storage Queue ops in cents) — OQ-1 validates at the first bill.
- **Bot traffic on public scale-to-zero ingress is unavoidable, and authentication cannot gate the wake** (verified 2026-07: ACA built-in auth runs as a per-replica sidecar — at zero replicas nothing rejects; the HTTP scale rule counts requests at ingress before any auth; Entra-side gates stop token *issuance*, and bots never ask for tokens; the only pre-replica filter is ingress IP restriction — useless for roaming phones). Accepted: one wake ≈ 75 vCPU-s against the 180k vCPU-s monthly free grant — cents even under scanning noise (OQ-1 watches the first bill; custom domains land in Certificate Transparency logs, so obscurity is not a plan). The guarded resource is Neon: **the unauthenticated path never touches the database** — JWT validation is local (cached OIDC metadata), the `ActiveAccount` lookup runs only after token validation, and **health probes stay DB-free** (Aspire defaults like to add DB checks — keep them off the probe path, M0). Entra "require user assignment" is enabled as an IdP-door mirror of FR-1 (stops uninvited humans, not bots). Contingency if the bill disagrees: Cloudflare free tier in front, ACA ingress IP-restricted to its ranges.
- Vendor lock-in to Azure is accepted deliberately for hosting/auth (it *is* the showcase); the database is the deliberate exception — standard Postgres on Neon stays portable (any Postgres host can take a dump/restore).
- Polyglot toolchain (dotnet + npm) in every pipeline.
