# DoseUp — Design workflow (Claude Design ↔ repo)

**Status:** decided 2026-07-18 ([#81](https://github.com/jakubbohm/DoseUp/issues/81); capability facts live-verified that day) · **amended 2026-07-19** after Jakub's in-app workflow conversation with Claude Design (§ 6's re-verify discipline firing: it resolved the inheritance unknown — the design system is read at *generation time* — and replaced the fresh-project-per-iteration idea with one persistent workspace) · part of [conventions](README.md) — docs-first source of truth

How DoseUp turns UI intent into shipped interfaces with Claude Design (Anthropic's design tool). The mandate itself — every UI-heavy change gets a mockup + handoff before implementation — is delivery process ([ADR-0004-delivery-and-process § OpenSpec workflow rules](../adr/0004-delivery-and-process.md)); this doc is the how. Claude Design is a research preview and moves fast: capability statements below carry their verification date and are **re-verified on first use inside any implementing change** — a hosted preview exposes no version signal to gate on — and whenever observed behavior deviates from this doc (the [software-factory F-39](../software-factory.md) discipline).

## 1. Terms — what Claude Design's words mean (verified 2026-07-18)

- **Project** — one design workspace (chat + canvas + a real file tree of design files). DoseUp uses **one persistent workspace project** for all design work, organized by feature folders (`features/<feature>/Screen.dc.html`) — the in-app conversation (2026-07-19) established that the design system is read at generation time (next bullet), so a persistent workspace always designs against current state and keeps earlier features as reusable context; a separate project only for throwaway explorations or if the workspace ever grows huge. The canvas has no rename/move UI — file management happens **by asking in chat** (the agent fixes cross-references when moving); give files descriptive names at creation so renames stay rare. Despite the similar name, this is distinct from the design-system project two bullets down.
- **Design system** — Claude Design's name for the reusable look-and-feel package: a "UI kit" of colors, typography, components, and layout patterns. **Mechanically a tokens + components library — not corporate branding.** Projects bound to it read it **at generation time** — every generation uses its current state; nothing is copied in at project creation (per the in-app conversation 2026-07-19; supersedes the earlier at-creation-inheritance reading). For DoseUp it holds exactly the token set and components the app builds on. An account can hold several; the workspace project must be **bound** to DoseUp's, and setting DoseUp's as the account **default** covers any future project (binding/default mechanics on § 6's confirm list).
- **Design-system project** — the dedicated project type (`PROJECT_TYPE_DESIGN_SYSTEM`) holding that package. Ours is **"DoseUp"** (created 2026-07-18, id `6ada98c2-bda4-4278-9995-50cee0d099db`). The empty "Design System" project left over from onboarding is never used.
- **Handoff (bundle)** — the app's "Hand off to Claude Code" export: a per-feature **spec bundle** — stack-agnostic HTML design references plus the design chat and a README recording layout, tokens, exact values, interactions, and states. The README instructs the implementer to recreate the design in the target codebase's own environment and patterns, so repo conventions govern implementation and the stack is never restated at handoff time (CLAUDE.md already carries it — in-app conversation 2026-07-19). Delivered as a pasteable prompt carrying the bundle URL; **scoped to the feature** — a full-project export only for the greenfield drop (afterwards it would carry stale copies of shipped screens and invite unrequested changes). This is the **only** channel design work travels back to us — Claude Design cannot write to the repo (the claude.ai GitHub integration is read-only, file contents on one branch).
- "Template" is not a first-class concept in the current product; this workflow doesn't use the word.

## 2. Source of truth & direction of flow (capabilities verified 2026-07-18)

**The repo is the single source of truth for tokens and components.** The DoseUp design-system project is a derived, pushed copy — never hand-edited in the app. Remix explorations (the app's duplicate-and-iterate action) on the canvas are fine, but their outcome returns through a handoff like any other design work.

- **Up (repo → Claude Design):** Claude Code's `/design-sync` (beta; its engine is the `DesignSync` tool, confirmed present in the 2026-07-18 local build) converts the client's component library into the uploaded design-system bundle, so the design agent builds with our real parts. What it extracts is the durable visual language actually implemented — tokens, fonts, reusable components, and *representative screen recreations*: a UI kit, not an exhaustive page inventory (in-app conversation 2026-07-19). It is **push-only** — it never pulls design content into the repo; third-party "two-way sync" wording means push-up + handoff-back, not bidirectional file sync (official copy never uses the phrase). It requires a *buildable* library (component source + tokens/CSS + compiled output, or a Storybook) and pins the target project in `.design-sync/config.json` (committed — our convention, § 4). Re-push after a merged change to tokens/components (§ 3 step 5).
- **Down (Claude Design → repo):** the handoff bundle only, consumed by an OpenSpec change. **Bundle content is input, never code:** the implementing change translates it into our conventions; nothing from a bundle is committed verbatim; generated agent-config files (a `CLAUDE.md`, rules, skills), lockfiles, or scaffolding inside a bundle are discarded on sight. **Translation maps to existing tokens and components** — a genuinely new token or component a handoff introduces is flagged in the change as a reviewable decision, never silently invented (the anti-drift rule from the in-app conversation 2026-07-19; mirrored as an openspec design rule).

## 3. The loop (every UI-heavy change)

1. **Brief** — a versioned prompt doc in `docs/design/` (first: [claude-design-prompt-first-screens.md](../design/claude-design-prompt-first-screens.md)). Briefs carry the aesthetic direction and may state the implementation stack — Tailwind CSS v4 ([ADR-0001-platform-and-stack](../adr/0001-platform-and-stack.md)) — as steering for token discipline, but the designs produced are **stack-agnostic HTML references** either way (in-app conversation 2026-07-19): design-time fidelity comes from the pushed design system (step 5), implementation-time fidelity from repo conventions — the handoff README defers to the target codebase, and CLAUDE.md carries the stack (§ 1). Component behavior is **Radix Primitives** (decided 2026-07-19 — [ADR-0001-platform-and-stack](../adr/0001-platform-and-stack.md)), always behind our own named components (§ 4); the data & state half of [#26](https://github.com/jakubbohm/DoseUp/issues/26) stays open.
2. **Design** — Jakub runs the brief in the persistent workspace project (§ 1) and iterates on the canvas; the work lives in that feature's folder. **Designs are snapshots** — they don't restyle themselves when the design system moves. So an update to an existing screen starts from the design system's synced recreation of the *shipped* screen (production truth, including whatever drifted during implementation), never by restyling the old design file; a screen missing from the synced kit is recreated first from its actual source (paste or attach the component code), then redesigned — after a sync, where a screen was originally designed no longer matters. Manual app-side acts: clicking the handoff, optionally linking the GitHub repo read-only for context, and the one-time binding/default act (§ 1).
3. **Hand off** — "Hand off to Claude Code"; the bundle URL enters the implementing session. Exports are **scoped to the feature** — the change id is the scope name — with a full-project export only for the greenfield drop (§ 1). The implementing change's tasks include archiving the bundle's README (its decisions record) under `docs/design/handoffs/<change-id>/`, linked from the issue ([project-management.md § 12](project-management.md) attachments rule). The bundle's design files/binaries themselves are never committed — a deliberate carve-out from § 12's commit-visuals convention: a mockup needed on an issue is exported and committed as a curated image, never as the bundle blob.
4. **Implement** — an OpenSpec change lands the translated tokens/components/screens in the client (mapping to existing tokens/components, new ones flagged — § 2); all other conventions apply unchanged.
5. **Re-sync** — if tokens or components changed, the change's tasks include an explicit **post-merge** `/design-sync` push step, so the next iteration designs with current parts — the workspace picks the push up automatically, since the design system is read at generation time (§ 1). Same task-rule mechanism as the TS-client-regen rule, one deliberate difference: regen commits artifacts into the same PR, while the push mutates the shared design-tool copy — so it runs from `main` after the merge (archive time is the natural moment), never from an unmerged branch.

## 4. Where design lands in the repo

- **Tokens: one file** in the web client (`styles/tokens.css`; exact path fixed by the scaffold change — [#43](https://github.com/jakubbohm/DoseUp/issues/43)): CSS custom properties with semantic names. Tailwind v4 consumes them CSS-first via `@theme` — static tokens become utilities; runtime-varying tokens (the ambient daypart hue) stay plain custom properties that utilities reference.
- **Components:** co-located in the client per the scaffold's layout; every reusable element is a named component (the brief's component list is the seed).
- **Styling is utility-first:** Tailwind classes wherever they express it, bespoke CSS only for what utilities can't say (the brief's cutting-edge effects); briefs restate this as steering.
- **The primitive wall:** feature code never imports the primitives library — only the owned component layer does (ESLint `no-restricted-imports`, wired by the scaffold change; the TS analog of the ArchUnitNET dependency rules). Keeps Radix swappable for its named fallback (Base UI — [ADR-0001-platform-and-stack](../adr/0001-platform-and-stack.md)).
- **`.design-sync/config.json`:** written by the first `/design-sync` run; verify the pin equals § 1's project id, then commit it.
- **`docs/design/`:** briefs + handoff records.

## 5. Bootstrap order (M0/M1 — greenfield exception)

Until the client exists there is nothing to push, and iteration 1 has no prior push to design from — the push that normally precedes an iteration (the previous change's step 5) happens here only after the scaffold change lands: first-screens run in the workspace project ([#58](https://github.com/jakubbohm/DoseUp/issues/58) executes it) → handoff (**the one full-export drop** — everything is new; scoped exports thereafter, § 1) → the scaffold change lands tokens + base components in the client shell ([#43](https://github.com/jakubbohm/DoseUp/issues/43)) → **first `/design-sync` push** → Jakub binds the workspace to the DoseUp design-system project and sets it as the account default (§ 1). From then on the steady-state loop above; the bootstrap is complete when the DoseUp project holds the pushed tokens and the shell references them. (Pre-creating the design-system project was optional — `/design-sync` can create one from the codebase — but ours exists and stays the pin.)
