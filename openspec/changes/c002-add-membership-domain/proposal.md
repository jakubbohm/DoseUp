# c002-add-membership-domain — Proposal

**Milestone:** M0 — Walking skeleton on Azure · **Serves:** FR-1 (partial — the account model complete-signup will create), FR-2 (foundations — the Disabled state and its rules; the admin slice is M3), FR-3 (partial — the account as the future profile owner) · NFR-4 (the ring-1 lookup key), NFR-7 (process)

**Tracks:** #53, #55, #97, #98 (closes on merge)

## Why

M0's walking skeleton hinges on the Membership module existing: the *get account detail* slice (#54/#42) needs an account to read, and `ActiveAccount` resolution (#47) needs the account row it resolves Entra's `oid` against. Today only c001's bootstrap placeholder (`DoseUpDbContext`, empty `Initial` migration) proves the schema pipeline. This change births the module — the Account aggregate as pure, unit-tested domain code, and the `membership` schema as the first real, module-owned migration — and retires the placeholder (#55), whose continued existence would otherwise force two contexts through the migration runner for nothing. It is also the first real subject for the domain conventions (typed ids, SmartEnum, `CheckCanXxx` rules, `DateTimeOffset now` parameters) — the showcase goal made concrete.

## What Changes

- **New module `Modules/Membership`** with declared rigor grade: full domain discipline, small surface.
- **Domain:** `Account` aggregate — `AccountId` (typed id, client-generated Guid v7), `EntraObjectId` (typed external-identity reference; the ring-1 lookup key), mandatory `DisplayName` and `Email`, `AccountStatus` SmartEnum (`Active = 1`, `Disabled = 2`, append-only), `CreatedAt`. Factory `Account.SignUp(oid, displayName, email, now)` lands `Active` — the recorded OQ-5 default, not a closure of OQ-5. Transitions `Disable()` / `Reactivate()` guarded by static-first `CheckCanDisable` / `CheckCanReactivate` (violation codes `account.not-active`, `account.not-disabled`). No domain events (nothing reacts yet); no admin anything on the entity — a permission model is a future change (decided 2026-07-19).
- **New SharedKernel union `DomainResult` (steering 2026-07-19 — the domain half of #38):** domain methods never return the edge `Result` union — its REST-taxonomy cases (`NotFound`, `Forbidden`, `Validation`, `Unexpected`) are nonsense inside an aggregate. `DomainResult` (`Success | RuleViolations`) is the domain layer's result; the aggregate's re-assert converts from `RuleCheck.Fail`, the handler maps to the edge `Result` via one SharedKernel conversion. Both forms ship: the non-generic (void-shaped operations — `Disable`/`Reactivate`) and `DomainResult<T>` (`Success` carries the operation's value) — the generic has no consumer in c002 (`SignUp` is rule-free by design: guards throw, the uniqueness set rule lives in handler + constraint), so shipping it now is a recorded, Jakub-directed exception to the SharedKernel growth rule — the pair is complete the day the pattern is taught, first consumer expected in M1. Domain stays sync by convention, so no `Task` variants at this layer. A new arch rule bans the edge union from Domain namespaces (#97 — the rule ships in the same PR as the type it enforces, per the conventions-and-enforcement-together rule). **#38 gets a comment, not closure** — the handler/endpoint half stays parked for its own call-site revisit.
- **The edge union is renamed `Result` → `ApiResult` (#98):** with the domain getting its own result type, the bare name `Result` no longer says which layer owns it — the pair `ApiResult` ↔ `DomainResult` tells them apart on sight. Pure rename, zero behavior/HTTP-contract change: the type, its file, the Platform mapper, tests, and every mention in the main specs and conventions docs (incl. CLAUDE.md and the `openspec/config.yaml` context blurb); ADR mentions stay historical per the issue's scope. #38's shape question is unaffected and stays open.
- **Persistence:** `Modules/Membership/Infrastructure/Persistence` — `MembershipDbContext` (schema `membership`, own `__EFMigrationsHistory`), `AccountConfiguration`, design-time factory, first real migration; **unique index on the Entra object id** (set-rule backstop for "one account per identity"); **snake_case identifiers per the decided convention** (F-88, [conventions/README.md § Persistence](../../../docs/conventions/README.md), 2026-07-19) via `UseSnakeCaseNamingConvention()`.
- **Spike #93 outcome baked in (GO, 2026-07-19 — evidence in `spikes/snake-case-naming`, merged via #96):** `EFCore.NamingConventions` 10.0.1 is proven on our exact preview pins. The migration authoring follows the three spike-learned rules — the package renames only *generated* names, and all three violations fail silently or worse: (1) `HasDefaultSchema` authored lowercase (an explicit `"Membership"` would survive verbatim and quote into every statement); (2) check constraints authored snake_case in **both** name and expression (the idiomatic spelling emits DDL Postgres rejects — and the set-rule backstop maps violations by constraint name); (3) EF's history table explicitly renamed, or it stays a quoted `"__EFMigrationsHistory"`. These rules are recorded in [conventions/README.md § Persistence](../../../docs/conventions/README.md) alongside the GO (mirroring how ADR-0002 recorded the Wolverine spike must-dos).
- **Placeholder retired (#55):** `Platform/Persistence` bootstrap context, its `Initial` migration, and its factory deleted; `Program.cs` and the `MigrationWorker` hard-wire `MembershipDbContext` — the per-module registration shape (d18, #39) stays open until a second module makes generalization real.
- **Convention naming refresh:** rule-check methods are `CheckCanXxx` (decided 2026-07-19); [domain-rules.md](../../../docs/conventions/domain-rules.md) examples updated (`CanEdit` → `CheckCanEdit`). Zero code impact — no `CanXxx` exists anywhere yet. Projected DTO affordance flags stay `canEdit`-shaped on the wire.
- **Admin-trace cleanup (decided 2026-07-19):** admin is neither an account state nor an entity flag. Amend [ADR-0002 § Authorization](../../../docs/adr/0002-architecture-style.md) ring 1 (`CallerContext` carries `AccountId` only; "account status is a database column"; the admin gate's backing store becomes the future permission-model change; the surviving principle: lifecycle state lives in-app, never in Entra app roles), mirror in [conventions/README.md § Authorization](../../../docs/conventions/README.md) and the `openspec/config.yaml` context blurb; rename the authorization-matrix caller class `revoked` → `disabled` (ADR-0002 § Testing, [testing.md](../../../docs/conventions/testing.md) § 4) to match the domain vocabulary; fix issue #53's body ("(active, admin)" → "(active, disabled)") — tracker mutation, on plan sign-off.
- **Stryker deferral recorded:** [testing.md](../../../docs/conventions/testing.md) § 7 scheduled the mutation-testing spike into this change; it is explicitly deferred — the scheduling note in testing.md/roadmap backlog is updated to say so, keeping the docs honest.

## Capabilities

### New Capabilities

- `membership-accounts`: the Account aggregate's behavioral contract — signup lands an active account carrying its Entra identity link, display name, and email; the Active ⇄ Disabled state machine with its rule checks and stable violation codes; identity-link uniqueness guaranteed at the schema level; state persisted in the module-owned `membership` schema.

### Modified Capabilities

- `shared-kernel-primitives`: ADDED requirement — the `DomainResult` union (`Success | RuleViolations`), in both void and value-carrying (`DomainResult<T>`) forms, as the domain layer's result type, with the `RuleCheck.Fail` → `DomainResult` and failure-side `DomainResult` → `ApiResult` conversions; domain code never returns the edge union. MODIFIED (rename-only) — existing requirements' `Result` mentions become `ApiResult` (#98).
- `error-contract`: MODIFIED (rename-only) — the mapper requirement's "Result-case → status matrix" wording follows the `ApiResult` rename; taxonomy, statuses, and the 409 contract are behaviorally unchanged.

`api-shell` is untouched (shared-kernel-primitives explicitly leaves member naming design-level, so `CheckCanXxx` is no spec change). The placeholder context was never specced — its removal is implementation.
